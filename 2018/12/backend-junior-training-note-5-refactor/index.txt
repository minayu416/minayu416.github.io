<h3 id="what-is-refactoring">What is Refactoring?</h3>
<p>根據<a href="https://zh.wikipedia.org/wiki/%E4%BB%A3%E7%A0%81%E9%87%8D%E6%9E%84">維基百科</a>所說，「重構」是在不影響程式輸出/ 結果的前提下，整理程式碼，讓其可讀性提高。</p>
<p>讓原本雜草叢生，凌亂不堪的程式碼，重構過後，變成整齊，可閱讀的文本。</p>
<p>還是新手熟悉語法，並不會學習到這個概念，等到與前輩們，團隊合作後，經過前輩的教導，接觸了重構。</p>
<pre><code>你可能會覺得你已經渾身解術寫出能動能跑的程式就很開心了
</code></pre>
<p>但真的要增進功力以及讓團員跟你合作美好</p>
<p>您必須讓團隊合作的團員看得懂你寫的程式碼，可以讓人一目瞭然，也可以讓人迅速抓出臭蟲(Bugs)，這是當你熟悉程式語言後，即將要學習的新觀念。</p>
<p>來貼張我以前寫的程式給大家欣賞</p>

<p>乍看之下，好像沒有我想像中想呈現的樣子，但依這張圖片來看，裡面包著三層 For 迴圈加上 if/ else，看起來沒有很複雜，但除錯的時候會很痛苦，你得一層一層檢查看變數有沒有帶下去，萬一遇到 6, 7, 8 個 for 迴圈再一起時，一定會瘋掉的!</p>
<h3 id="why-refactoring">Why Refactoring?</h3>
<p>根據書上列出的四大優點</p>
<ol>
<li>
<p>重構使改進軟體設計</p>
</li>
<li>
<p>重構使軟體更易被理解</p>
</li>
<li>
<p>重構助你找到臭蟲</p>
</li>
<li>
<p>重構助你提高編成速度</p>
<p>Why Refactoring Works — Kent Beck</p>
<p>程式有兩面價值「今天可以為你做什麼」「明天可以為你做什麼」</p>
<p>If you emphasis today’s code, you can work with this code tomorrow.</p>
<p>If you found that today’s code is not suit for tomorrow, you have to make some choose.</p>
</li>
</ol>
<p>是什麼讓程式難以相與？ 有四個原因</p>
<ol>
<li>
<p>難以閱讀的程式，難以修改</p>
</li>
<li>
<p>邏輯重複(duplicated logic)的程式，難以修改</p>
</li>
<li>
<p>添加新行為時需要修改既有程式碼者，難以修改</p>
</li>
<li>
<p>帶著複雜條件邏輯的程式(complex conditional logic)，難以修改</p>
</li>
</ol>
<p>因此，我們希望程式</p>
<ol>
<li>
<p>容易閱讀</p>
</li>
<li>
<p>所有邏輯都只在唯一地點指定</p>
</li>
<li>
<p>新的改動不會危及現有行為</p>
</li>
<li>
<p>盡可能簡單表達條件邏輯(conditional logic)</p>
</li>
</ol>
<h3 id="when-refactoring">When Refactoring?</h3>
<h3 id="時時牢記的三條規則-the-rule-of-three">時時牢記的三條規則 (The Rule Of Three)</h3>
<ul>
<li>添加功能時一併重構</li>
<li>修補錯誤時一併重構</li>
<li>複審程式碼時一併重構</li>
</ul>
<h3 id="bad-smell-程式碼的壞味道">Bad Smell 程式碼的壞味道</h3>
<p>經過上述簡單將 What, Why, When 交代清楚了，但你未必知道哪些程式碼需要重構，因此你需要懂得分辨程式碼的壞味道，哪些程式碼一看就知道需要重構，在這我不會分享所有的分類，我會介紹書中幾個技巧配上我的範例介紹給大家</p>

<h4 id="large-class-過大類別">Large Class (過大類別)</h4>
<p>過大類別就是制定一個 Class 物件，Class 為因應不同的物件所需會有不同的函式時，而延生出一個 Class 裡有一堆不同的函式</p>
<p>我已經不知道我在說什麼了，但目前在寫的程式跟過大類別相似，卻不知該如何分享</p>
<p>只好想像有個物件是判斷<strong>貓科(Felidae)動物</strong>，可是貓科動物有很多不同的動物，總不能將所有跟判斷貓科的條件都寫在同樣一個 Class 裡，這樣會造成這個 Class(判斷)，會包含很多函式(條件, 依據)</p>
<pre><code>Class &#x3C;strong>Felidae(object):
    def legs():
       ....

    def hair():
       ....

    def color():
       ....&#x3C;/strong>
</code></pre>
<p>可以設定<strong>SubClass</strong> 子類別或超類別來幫助判斷</p>
<pre><code>Class &#x3C;strong>Felidae(object):

    @abstractmethod
    def criteria():
        pass

    @abstractmethod
    def rule():
        pass

    @abstractmethod
    def must_have():
        pass&#x3C;/strong>


Class &#x3C;strong>Cat(object):
    def criteria():
        ....

    def rule():
        ....

    def must_have():
        ....&#x3C;/strong>

Class &#x3C;strong>Lion(object):
    def criteria():
        ....

    def rule():
        ....

    def must_have():
        ....&#x3C;/strong>
</code></pre>
<p>可以用 Extract Method 或 SubClass 來幫助避免過大類別發生</p>
<p>雖然定義的有點簡陋 XD 但有把意思解釋到比較重要</p>
