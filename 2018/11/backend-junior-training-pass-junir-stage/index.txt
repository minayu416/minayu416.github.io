9:T1e01,<p>在背景知識不太夠，程式技術也才剛起步時</p>
<p>新手通常是利用關鍵字查找 Google</p>
<p>查到什麼就下什麼指令或跟著做，幸運的話就可以動</p>
<p>不幸運的話，頂多就是換下個查找到的網頁繼續試</p>
<p>就這樣一步一步將程式建構起來</p>
<p>這個階段通常是剛學程式的歡樂驚喜幼幼時期之</p>
<p>程式語言好玩好玩真好玩時期</p>
<hr>
<p>但這一切在我進公司後，有了很大的化學變化</p>
<p>面對新的專案或者公司內部開發的產品</p>
<p>你會不知道該如何下手，網頁應該查不到</p>
<p>畢竟專案/產品可能是在幼幼時期的好幾倍困難</p>
<p>即便是從資訊碩士剛畢業的自己，也覺得業界與學校的 gap 挺大</p>
<p>剛進公司開始我還是照著以前的方法報什麼錯就拿去辜狗，並且亂搞一通</p>
<p>從來沒問過自己</p>
<p>除了抄程式碼外，我也從來沒去了解我抄的程式碼背後的意義</p>
<p>隨著在公司陸陸續續的進步</p>
<p>我想分享一下，怎麼突破幼幼時期的門檻，繼續晉升前進</p>
<p>在前幾篇文章中，我有分享前輩給我制定了一項訓練計畫</p>
<ul>
<li>初階工程師技能</li>
<li>專案相關技術學習</li>
</ul>
<p><strong>初階工程師的技能分別是</strong></p>
<ol>
<li>
<p>學習 debug 工具，並運用在追蹤程式/問題</p>
</li>
<li>
<p>開始接觸 Design Pattern</p>
</li>
<li>
<p>學習 Unit Testing，並掌握 3A: Arrange, Act, Assert 的開發節奏</p>
</li>
<li>
<p>學習 重構 refactoring，並能識別 bad smell</p>
</li>
</ol>
<p><strong>專案相關技術學習</strong>其實就是要了解這個專案要做的功能會用到的技術</p>
<p>好比如果是做資料庫的 CRUD, 可能就會需要學習 SQLAlchemy</p>
<hr>
<p>根據上面的兩項： <strong>初階工程師技能</strong>及<strong>相關技術學習</strong></p>
<p>若說在「資訊」相關學系學到的屬後者：<strong>相關技術學習</strong></p>
<p>**「相關技術學習」**也可以說是程式語言＋背景知識的融合</p>
<p>畢竟寫程式是為了解決問題，在這個情況下就能得知為什麼程式語言就只是工具</p>
<p>就算程式語言本身還沒有學到很熟，也可以邊開發專案邊學習</p>
<p>所以簡單來說當我們今天開發一個需要連結資料庫做 CRUD(Create, Read, Update, Delete)操作及對外接收資訊時</p>
<p>會需要用到的相關技術以 python 來舉例，就會是</p>
<ol>
<li>
<p>先去了解開發的功能會需要用到哪些<strong>背景知識</strong>(資料庫, 如何做接收參數)</p>
</li>
<li>
<p>熟悉自己使用的程式語言相關的套件與語法</p>
</li>
</ol>
<ul>
<li>資料庫: pymysql, sqlalchemy</li>
<li>接收資訊: request, (懂 json 格式，或以 list, dict 做傳遞)</li>
</ul>
<p>以上是**「相關技術學習」**</p>
<hr>
<p>初學程式時，「函式」對我來說是較難理解的部分</p>
<p>我會想盡辦法避開它，所以我寫的程式會越寫越落落長</p>
<p>變得不易去讀，也難除錯</p>
<p>當開發小型專案時或許能這麼做，但是當需要用到兩個甚至三個以上的 for 迴圈，你就會很頭痛</p>
<p>光是學習程式語言及背景知識其實還不夠</p>
<p>接下來來講講「<strong>初階工程師的技能」</strong>，是我進公司才開始學習</p>
<p><strong>初階工程師的技能</strong>，主要功用是輔助自己<strong>寫好</strong>程式</p>
<p>當剛進入一個開發到一半的專案，或者你遇到 bug 欲解決時</p>
<p>除了看錯誤訊息外，我們在<a href="/2018/11/%E5%BE%8C%E7%AB%AF%E5%B0%8F%E8%8F%9C%E9%B3%A5-debug-%E6%B5%81%E7%A8%8B/">此篇</a>有介紹 Debug 的流程</p>
<p>突破幼幼班時期的第一個：學會使用 Debug 工具</p>
<p>除了拿來利用 Debug 外，也可以拿來追蹤整體的 Code</p>
<p>各大編譯都有 Debug 工具，我寫 Python, 所以我用 Pycharm (可是我一開始用 Atom, 然後變成 VScode, 最後才是 Pycharm)</p>
<p>我覺得 Python 的 Debug 工具 必學 Step Over, Step Into, Step...</p>
<p>以及利用 Debug 工具開始去了解每一行程式的功能，傳遞了什麼參數，做了什麼</p>
<p>再也不是查網站亂抄程式，出錯看錯誤訊息，將訊息貼到估狗查詢，亂處理一通了(這為我的慘痛經驗)</p>
<hr>
<p>第二：學習 Design Pattern</p>
<p>在網路上有篇網誌比喻 Design Pattern 為遊戲的連續技</p>
<p>在學校我完全沒有聽過這個名詞，當時只知道程式語言要好好的學習，完全無知 Design Pattern</p>
<p>Design Pattern 學習連結 ：<a href="https://openhome.cc/Gossip/DesignPattern/">非關語言：設計模式</a></p>
<p>當認識程式語言後，撰寫過許多次程式語言後</p>
<p>過了一兩年，有些人回來看自己的程式碼，也許會覺得怎麼這麼慘不忍睹 XD</p>
<p>你也曾寫過那種落落長長一行寫到底毫無分段分層的程式碼嗎？</p>
<p><strong>我有 XD</strong></p>
<p>Design Pattern 也是前輩認為突破幼幼班的關鍵之一</p>
<p>剛學習寫程式，一定有很多人會寫出很長很長，完全不換氣的程式碼</p>
<p>往往會造成很難閱讀，甚至當出 bug 時，也找不到 bug</p>
<p>尤其剛學習程式，也不太會 debug，就這麼的愣在那邊</p>
<p>附上的網站 title 就是非關程式，除了程式語言與背景知識外</p>
<p>其實軟體開發還需要學習很多跟以上兩者無關卻又非常重要的知識</p>
<p>慢慢學習 Design Pattern 讓你了解前輩們的程式碼，也能讓自己寫出一口好閱讀的程式碼</p>
<hr>
<p>第三：Unit Testing</p>
<p>在近代專案開發漸漸往「敏捷開發」前進</p>
<p>好險我在研究所的專案管理有讀到敏捷開發的論文</p>
<p>進公司後能夠快速了解公司的專案流程與迅速得知敏捷方法</p>
<p>早期專案通常是將全部程式都寫完後，才做測試</p>
<p>但往往會發現一堆 bug, 這時候專案也近尾聲，讓工程師們急著趕工好像也不好</p>
<p>除了讓專案 bug 一堆就完工外，就是那大群程式碼不知道怎麼找 bug</p>
<p>在敏捷方法中有一個方法叫 <a href="https://zh.wikipedia.org/wiki/%E6%B5%8B%E8%AF%95%E9%A9%B1%E5%8A%A8%E5%BC%80%E5%8F%91">TDD</a> , Test-driven development , 中文叫 「測試驅動開發」</p>
<p>主要是希望在撰寫程式前先寫好測試，再寫程式</p>
<p>功用是希望能在撰寫程式的同時做好測試，這樣能夠立即知道 bug 出在哪</p>
<p>書本上其實是顯示，先測試後寫程式的方法主要是因應客戶提出的需求</p>
<p>先在測試中寫出客戶的需求，然後之後寫的程式要能符合測試上的需求</p>
<p>我覺得這個算比較進階了，我也還在學習，但這依然是軟體開發中重要的技能</p>
<hr>
<p>最後一個前輩列出的是：重構 refactoring</p>
<p>重構我也還沒去做學習，這邊不好去解釋什麼</p>
<p><a href="https://zh.wikipedia.org/wiki/%E4%BB%A3%E7%A0%81%E9%87%8D%E6%9E%84">程式碼重構</a>，貼了一下維基百科</p>
<p>基本上功用是在不會讓軟體掛掉的前提下，將程式碼整理整理，讓可讀性變高</p>
<p>功能其實是</p>
<ul>
<li>讓可讀性變高</li>
<li>減少 bug 產出</li>
</ul>
<p>有一句話是</p>
<p>善用 Unit Test + 重構</p>
<p>可提高開發效率，讓程式可讀性變高，減少程式產生 bug，並在測試時符合需求，也在測試時提早抓出 bug</p>
