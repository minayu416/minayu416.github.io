<p>在昨天遇(ㄉㄨ ˇ)到指導老師後</p>
<p>就感覺太陽出來了</p>
<p>喔不是，是又有了新一步的發展</p>
<p>自己卡了幾天後，除了慌張以外還是找學弟們詢問</p>
<p>不過昨天遇到指導老師後，也得到了幾條線索，拿回家繼續摸索</p>
<p>首先，觀念釐清了</p>
<p>這次論文要開發的程式是</p>
<p>Python Desktop Application(for Windows)</p>
<p>主要功能不能透露</p>
<p>會將資料利用 Restful API 經過 HTTP/ Django Requests 回傳虛擬機器(雲端)</p>
<p>而問完指導老師後，釐清的第一個觀念是</p>
<p>發送端及接收端=</p>
<p>專有名詞，沒這專有名詞還真不知道該估狗什麼 QQ</p>
<p>Client 端: Python Desktop Application</p>
<p>Server 端: Vm/ Django Requests</p>
<p>使用技術：</p>
<p>Python, Django, Restful API framework(concept), ...</p>
<p>使用設備：Atom, Terminal, Mac Air, VM, MySql, ...</p>
<hr>
<p>以往只有開發網頁經驗的自己，對於開發桌機一開始還蠻恐懼的</p>
<p>覺得能夠寫桌機程式是跟神一般的存在</p>
<p>尤其還要將桌機的資訊回傳給網路</p>
<p>然後一開始很緊張的時後一直跟人家再三確認，桌機真的能夠回傳給網路？？？</p>
<p>一直問一直問一直問 XDDDD</p>
<p>不過到目前已經了解很多概念跟技術</p>
<p>也就不會這麼慌</p>
<p>寫程式最重要的除了邊寫邊學之外</p>
<p>就是在寫程式之前，簡單的定義好事項，像是</p>
<ul>
<li>功能</li>
<li>需求</li>
<li>架構</li>
<li>用到什麼技術</li>
<li>需要什麼設備</li>
</ul>
<p>因為程式很複雜，尤其老師給的需求跟講的概念每週都不同。</p>
<p>你一定也會因為卡住看了一堆網頁，然後自己亂掉。</p>
<p>我自己是會在每次開會後，先把功能跟需求還有系統的大約架構用 google 雲端上的 diagram 簡單畫一下，也算是總體架構，讓自己看了也比較知道要做什麼，才不會亂掉</p>
<p>這部分我們老師常常也都扔給我做，他去寫程式，寫好的程式帶出來給我畫架構給別人看</p>
<p>目前自己要開發的程式分為兩端</p>
<p>也就是</p>
<ul>
<li>Client 端</li>
<li>Server 端</li>
</ul>
<p>而 Client 端目前做好的是兩個子系統 兩個 python .py 檔(有一個還得我自己拆出來)</p>
<p>而需要做的內容就是寫一個將兩個.py 檔可以執行起來然後將蒐集的數據透過點擊按鈕之類的動作回傳網路</p>
<p>(也就是說這部分到最後會變成一個 exe 檔，執行>蒐集資料>回傳)</p>
<p>而 Server 端目前是利用架在虛擬機上的 Django 接收檔案</p>
<p>因為 Django 有內建 Requests 語法，可以接收從客戶端來的檔案，並將他們回傳於資料庫及雲端裡，雖然是寫網頁，但能不能做到這樣，我也還在研究 QQ</p>
<p>而目前已經把資料庫準備好了，就再查怎麼寫 View 跟 Requests 了</p>
<p>而老師那邊是提出 HTTP Requests 的概念，還不懂是什麼，所以只能夠姑且再看看了 QQ</p>
<p>剛剛有稍微查一下網站，先筆記起來</p>
<p><a href="https://docs.djangoproject.com/en/2.0/topics/http/">https://docs.djangoproject.com/en/2.0/topics/http/</a></p>
<p><a href="https://docs.djangoproject.com/en/2.0/ref/request-response/">https://docs.djangoproject.com/en/2.0/ref/request-response/</a></p>
<p>這週進度，我就不講有什麼進度了，基本上早就天窗 XD</p>
<p>那我就來打一下剩下要做的事吧</p>
<p>Server 端：</p>
<p>將 view 接收的部分寫好</p>

<p>將其中一個 .py 子檔獨立出來</p>

<p>利用 Tkinter 整合兩個 .py 子檔</p>
<p>設計按鈕結束兩個子檔程式並回傳網路</p>
