9:T727c,<p>太陽能監控平台是我在上一家公司做的第二個專案/產品，在上一家公司的職涯生活，約有前 1/3 的時間在參與實作路燈的專案。然後之後的 2/3 生活都專注在太陽能產業軟體開發應用。</p>
<p>陸陸續續做了兩三個和太陽能能源相關的專案及產品，在這段職涯中，我擁有了許多酸甜苦辣的回憶，包括在路燈專案後的程式能力開始大幅成長。能力成長了，責任也變重了，所需要負責的事情也變多了，漸漸成長為一個獨當一面且有深度的人。</p>
<p>在這篇文章中，我只會介紹與太陽能監控平台與資料數據分析相關的經驗分享，其他一些比較小的專案就不列於此。</p>
<p>當時有參與實作太陽能監控相關專案的，除了 <code>太陽能監控平台</code> 之外，後來還獨立開發一個效能比較高的數據收集程式，用了比較深的技術跟較精簡的程式碼。</p>
<h1>介紹 | 緣起</h1>
<p>2020 年 5 月，路燈專案確定暫時中止開發時程，下一個案子是決定和同層樓的太陽能維運及業務部門合作，開發一個一頁式太陽能監控平台，然後掛在大電視上供人觀看。</p>
<p>起源其實是同時有另外兩家太陽能監控公司在實作太陽能監控平台，且其中一家公司的專案目前就擺在電視上。但當時的主管跟其他部門的主管希望製作網站的掌控權由我方掌控，因此在路燈專案暫時終止後，我們主管跟專家們就希望透過我們開發團隊的力量來開發一個自己製作的太陽能監控網站平台。</p>
<p>而這次是真的開始有在和 IoT 裝置的即時巨量資料奮鬥，真的有來自 120 多個遍佈台灣的太陽能案場，共計 1000 多個太陽能感測器每兩分鐘回傳即時資料。所以開發出來的後端程式需要能夠乘載這樣的巨量數據流動。</p>
<p>短時間的目標是能夠做出取代電視上別家公司製作的一頁式監控網站。也就是說首要任務是先透過後端接收巨量資料，處理，計算並分析出來的數據，整合至一頁式網站，然後可以放在電視上，讓眾人一目瞭然目前全台灣太陽能案場的即時狀況。</p>
<p>這個網站不只是提供監控即時狀況，太陽能能源研究的學者或專家也能透過這個網站呈現的各個數據來檢視是否有發電效率低下或異常的狀態。而太陽能案場維運人員也能夠透過網站上的數據即時得知目前有無設備異常或損壞。</p>
<p>而中後期計畫就會是將原本一頁式網站擴展到一個管理網站。</p>
<p>會具備智能分析及深度分析的功能，提供報表及計算更深層數據的功能。而維運方面則是期望能透過軟體的條件判斷，將設備異常的感測器與太陽能案場裝置自動化開啟成報修單，開發新的報修功能，能讓廠商及維運人員合作修復損壞裝置。</p>
<p>深度知識方面，當初還研究到需要計算變流器的傾斜角度，搭配一日即時太陽日射角度來計算最大效率發電量。</p>
<p>IoT 裝置有趣的是，裝置由於會座落於各種戶外地點，極為不穩定，聽主管說當時有著裝置泡在水裡，有的經過日曬雨打，有的被有心人士破壞或偷竊及轉賣，一個工業級的裝置也花費不少金錢製作與購買，所以能夠實作與 IoT 相關裝置的軟體，已經是很新奇有趣也難得的體驗了。</p>
<p>也由於極其不穩定，所以必須要設計更多類似<code>解決時間差</code>，<code>訊息過濾</code>，<code>訊息遺失</code>，<code>重補資料</code>，<code>重新計算</code>，<code>異地備援</code>等其他解決方法來解決更多導致數據會有誤差的問題。</p>
<p>在這個產品，可說是學習到非常多，技術大幅成長，太陽能相關知識，軟體開發相關知識及許多商業價值的知識。</p>
<p>那接下來就來介紹本篇的重點: 太陽能監控平台與數據分析</p>
<h1>目標: 一頁式整合太陽能數據監控網站</h1>
<p><img src="https://mingjungyu.files.wordpress.com/2021/09/screen-shot-2021-09-21-at-4.07.39-pm.png" alt=""></p>
<p>我用 KeyNote 簡單化出了網站的架構，大概就是如圖，一頁式，然後放大放在電視機上。</p>
<p>來介紹網站中的各個部分含義，在這個網站中一共有六個部分，將從感測器收集的太陽能即時數據經過處理以及計算，最後輸出到這個網站中，一共分為：</p>
<ul>
<li>總覽 Summary</li>
<li>參考發電量/實際發電量</li>
<li>哪幾個太陽能廠產生最多電量 (日結算)</li>
<li>當地溫度 (即時)</li>
<li>營運狀態 (含六個狀態)</li>
<li>哪一個地區的太陽能廠</li>
</ul>
<p>由於只是 demo 示意圖，所以有些細節我沒有畫上去，每個數據都是每兩分鐘即時更新，下方三欄會變動，因為總共有 120 個太陽能案場，所以分成台灣不同的市區每 10 秒換地區呈現狀態。</p>
<p>左上方的 Summary 就是呈現所有太陽能廠累積加總的數據，所以像是</p>
<ul>
<li>總共累積發電量</li>
<li>目前營運的太陽能案場</li>
<li>目前維修的太陽能案場</li>
<li>發電效率及達成率</li>
<li>減碳 CO2 量</li>
</ul>
<p>還有更多指標沒有在上面呈現，加總大概有 8-9 項加總數據。</p>
<p>最中間的參考發電量及實際發電量看起來只有三欄指標分別為 (日/月/年的參考/實際發電量)，這一塊是整個網站中程式設計較為困難，計算上也較為困難的部分。</p>
<p>參考發電量: 是要經過一套數學公式將所有每兩分鐘的累加變流器數據加上每兩分鐘即時的日射數值及即時的溫度來去計算出來。</p>
<p>實際發電量: 就是計算目前實際的發電量。</p>
<p>還有一項指標是達成率，就是左邊%數的資訊，基本上就是 <code>實際發電量</code> / <code>參考發電量</code> 的百分比，就是一個計算出來的指標，顯示目前實際發電量發電達成了多少參考發電量。</p>
<p>這部分數據困難的是需要即時的資料外，還需要從每天早上 5 點太陽能廠開機開始的數據一直累積到即時的數據，所以必須從早上第一筆資料開始累計加總跟計算，漏了一筆就會有誤。</p>
<p>所以在規劃上需要做很多的防範措施以免萬一遺失資料，需要設計模擬推算出的資料取代為遺失的資料並標記為遺失，然後透過晚上回補資料的機制將遺失的資料補回。</p>
<p>右上方則是將當日發電量的太陽能廠由高至低排列(正確來說應該是按縣市的太陽能廠加總去比較)</p>
<p>下方的三欄會一起跳動，他們代表的是</p>
<p>左: 某個縣市的即時天氣
中: 該縣市的所有太陽能案場即時狀態(有六個狀態，像是溫度異常或者設備損壞)
右: 用地圖標示目前在哪個縣市</p>
<p>總共大約是用了 2-3 個月的時間將這個一頁式監控網站(Dashboard)給做出來，因為已經有別的公司做出正式運行的版本，所以大部分的時間都花在和正式運行版本對照，每個數字都需要準確，每個公式都要正確無誤，也就是說如果該公司私下有訂定一些規則去濾掉不規則的值或者一些其他的規則跟條件，我們是不知道，且要想辦法數字跟正式運行的網站一樣。</p>
<h1>太陽能案場內部裝置資訊</h1>
<p>其實這個部分是我最後才寫的，因為對於太陽能廠內部架構，一直到快離職前都還沒接觸到太多，畢竟在後端軟體工作的我們，只需要有個初步的概念，知道哪些裝置及感測器，它的功用大概是什麼，他會怎麼傳訊息跟傳什麼訊息給我們，大約略知即可。</p>
<p>雖然可能我畫的圖並不是最正確的，畢竟不是承包商也不是實際赴當地架設的人，但還是稍微簡略化一下架構並作說明。</p>
<p><img src="https://mingjungyu.files.wordpress.com/2021/09/screen-shot-2021-09-26-at-2.30.00-pm.png" alt=""></p>
<p>這個是<code>我目前腦海內</code>的太陽能廠裝置佈置圖，我知道漏了很多很重要的其他裝置，但後端的程式大部分只會接收來自這些裝置的訊息。</p>
<p>後端的程式透過 MQTT (本專案是透過 api 取得資訊，這個下面內容會描述)，取得的裝置內容為:</p>
<ul>
<li>變流器 Inverter</li>
<li>溫度計 Thermometer</li>
<li>日射計 Radiator</li>
<li>風向儀 Anemoscope</li>
<li>風速計 Anemometer</li>
</ul>
<p>之所以會寫太陽能板 Panel，是想示意説，真正的太陽能板是接在變流器後，而且中間還有其他裝置跟較複雜的串並關聯，不過這在不多做討論。</p>
<p><strong>變流器</strong>主要會傳遞發電量及和電能相關的參數，是計算即時發電量，累積發電量及其他重要與發電相關的資訊</p>
<p><strong>溫度計</strong>則是回報當地太陽能廠目前的環境溫度，只會有一個，代表當地的環境溫度，可提供搭配計算參考／預期發電量</p>
<p><strong>日射計</strong>則是回傳日射量，也只會有一個去測量太陽照射的量 (通常早上會比較少，直到中午跟下午，日射量可能會破萬，然後傍晚漸漸降低)，可提供搭配計算參考／預期發電量。開機時會傳遞不正常值，若發現不正常值需要濾掉。</p>
<p><strong>風向儀</strong>跟<strong>風速計</strong>目前還不了解參數內容，因為這兩個是非常後期加進來的裝置，直到我們要開始動工納入計算前，公司政策就改變了(這個下面也會提及)，主要回傳內容應該是與當地的風速及風向相關。</p>
<p><strong>太陽能板</strong>視擺放位置可能會有角度問題，像是傾斜度，所以其實在當時的未來規劃是希望能紀錄細項到太陽能板的傾斜角度，然後搭配溫度跟日射量還有變流器去計算出準確的參考／預期發電量，可以檢討為什麼實際發電無法達標。</p>
<p>最後雖然跟裝置無關，但是在當時的未來規劃中，也許要去即時計算太陽的仰角與傾角角度搭配其他數據去做計算。</p>
<p>總之無論是太陽能廠與太陽能能源相關的數據都是另外一門學問，而作為開發團隊的我們就是努力和專家了解這些資訊，盡量做出準確以及能夠協助使用者的好產品／專案。</p>
<h1>軟體架構</h1>
<p><img src="https://mingjungyu.files.wordpress.com/2021/09/screen-shot-2021-09-21-at-5.05.29-pm-e1632215164953.png" alt=""></p>
<p>看起來畫得有點複雜，讓我來解釋一下</p>
<p>原則上最終目標都是產出上面提及的 Dashboard，所以 Front-end 的部分就是 Dashboard 呈現的外觀，而後端的部分就是將螢幕上面所有的數據計算與產出給前端，也就是 <code>Main Process Program</code>跟<code>IoT Devices</code>的部分。</p>
<h2>前端網頁如何取得資訊?</h2>
<p>原則上這個 Dashboard 上的數據是每兩分鐘更新的即時數據，所以會有兩個部分維持能讓前端能維持取得最新的資訊。</p>
<ul>
<li>從後端網站取得</li>
</ul>
<p>若使用者有</p>
<ol>
<li>剛打開網頁</li>
<li>重整網頁</li>
</ol>
<p>那麼最新資訊就會是由前端去送 request 請求給後端，取得頁面上所有的即時資訊。</p>
<p>但問題既然是 Dashboard，他大部分的時間就會是一直開著網站，不會有任何重整或重開的動作。</p>
<p>這時候就需要規劃另外一種方法來取得最新資訊。</p>
<ul>
<li>訂閱 MQTT，即時接收後端每兩分鐘剛計算好的數據，透過 MQTT 傳給前端</li>
</ul>
<p>不只後端，前端的部分也有 MQTT 技術，因此為了要維持前端的頁面都是最新資訊，我們請前端訂閱<code>產出結果</code>的頻道，然後後端的程式則是每兩分鐘從感測器收集資訊，計算後將結果發送至<code>頻道</code>，這麼一來前端網頁就能即時獲得資訊。</p>
<h2>後端網站 (Back-end Web)</h2>
<p>在這個專案中，後端網站的部分只有簡單的一個功能，就是將計算好的即時資訊提供給前端，所以只開一個 api，而由另外一隻程式處理與計算的結果會存在 Redis 中，這隻 api 的功能就是從 Redis 讀取計算好的即時結果，然後回傳給前端網頁。</p>
<p>這麼一來無論是剛開頁面或刷新頁面，都能即時取到最新資訊。</p>
<h2>即時處理數據 (Data Processor)</h2>
<p>即時處理數據的程式，是這個 Dashboard 主要的後端程式開發，我們先稱呼他為<code>資料處理程式</code>，簡單來說就是每兩分鐘會驅動一次該程式，他會接收每個太陽能案場的感測器即時資料，處理並計算後，發送即時資訊到 MQTT 的指定頻道，提供前端網頁即時資訊，而另外則是將結果存入 Redis，讓後端網站可以透過讀取 Redis 的結果，提供最新資料給前端。</p>
<p>當然所有的處理與計算的結果或者原始資料都會紀錄於資料庫。</p>
<p><img src="https://mingjungyu.files.wordpress.com/2021/09/screen-shot-2021-09-22-at-12.36.00-pm.png" alt=""></p>
<p>接著來看看實際程式實作的架構圖，也就是說這個<code>Data Processor</code>的程式實作邏輯:</p>
<ul>
<li>先取得所有太陽能案場以及旗下的感測器裝置資訊</li>
<li>利用裝置資訊取得即時的感測器資料 (稍後說明)</li>
<li>處理資料
<ul>
<li>擷取重要資訊與數據</li>
<li>單位換算</li>
<li>去除不正常數據</li>
</ul>
</li>
<li>計算資料
<ul>
<li>統計無反應，損毀或不正常數據的感測器，並計算六種錯誤狀態</li>
<li>計算總發電量，累計發電，發電排名及 CO2 減碳量資訊</li>
<li>計算預期/參考的發電量，實際發電量與達成率</li>
</ul>
</li>
<li>儲存
<ul>
<li>將原始資料及計算後的數據儲存於資料庫</li>
<li>將初始化值，需更新的值及最終計算結果儲存於 Redis</li>
</ul>
</li>
<li>透過 MQTT 發布給前端</li>
</ul>
<h3>取得即時資訊</h3>
<p>看著架構圖跟描述會覺得有點奇怪，怎麼太陽能案場的感測器不是透過 MQTT 即時接收到後端程式？</p>
<p>在這邊說明一下，一個比較特別的點，由於上面有提及<code>公司先前已經有委託其他公司協助製作監控管理平台</code>，也就是說已經和某家公司協調，由他們包辦<code>太陽能場，地方所有的感測器佈置</code>以及<code>接收資料與儲存資料</code>，因此我們沒有辦法去設定感測器傳資料的設定，也無法知道 MQTT 的頻道，都屬另外一間公司的商業機密。</p>
<p>但可以接受我方透過對方的提供的後端 api 取得資料，所以算是二手資料，就是最開始已經是對方先接收處理過了，才會讓我們能夠過 api 取得資料，約莫會延遲 2-4 分鐘的資料。</p>
<p>也由於對方的後端程式設計，我方若需要透過 api 取得資料時，需要以<code>太陽能廠資訊</code> + <code>裝置名稱</code> + <code>取得時間</code> 做為參數，因此才會有第一步<code>取得裝置資訊</code>。</p>
<p>第一步驟就是先取得<code>太陽能廠及其每個感測器的資訊</code>，由於程式是每兩分鐘排程執行一次，總不可能每次都去資料庫要一次資料，再加上經過討論，裝置極少機率會需要更換或替補，不太會常更動，所以在<code>裝置資訊</code>的設計，我們將所有的裝置資訊儲存於<code>Redis</code>中，除非有新增刪除裝置，才會重新將資訊輸入 Redis，這麼一來每兩分鐘需要取得裝置資訊的動作也就不會這麼頻繁的對資料庫做操作。</p>
<p>第二個比較複雜一點，由於上述提及資料是在別的公司的資料庫中，需要透過 api 去取得，而需要用<code>資料時間</code>做為參數，來取得指定時間的感測器即時資料，由於怕資料過於龐大，所以一次只開放取 30 分鐘的資料。參數規格會是<code>2021 08/22 08:02 ~ 2021 08/22 08:32</code>，如果你以為這麼簡單的話，就太天真了！</p>
<p>因為必須考量到可能會有感測器的資料是遺失的 (而且常常發生類似停機或遺失的狀況)，所以在設計取資料的時間區段，就不會是以當下的時間加減 30 分鐘去取。我們會為每個感測器加上最新資料的時間然後存儲於 Redis，這是模擬格式:</p>
<p><code>solar:site:&#x3C;site_id>:&#x3C;sensor_id> : 1629616967</code> (換成 UNIX 時間，因為 Redis 不能存 Datetime)</p>
<p>而依據就是取得資料時，會取得 30 分鐘資料:</p>
<p><code>2021 08/22 07:22</code> ~ <code>2021 08/22 07:52</code></p>
<p>這只是取得的指定期間，實際上該感測器最新的資料時間可能只會到 <code>07:42</code>，那就需要紀錄目前該感測器的資訊只取到 <code>07:42</code>，然後下一次要透過 api 取得資料時，就會變成是</p>
<p><code>2021 08/22 07:42</code> ~ <code>2021 08/22 08:12</code></p>
<p>從<code>07:42</code>，最新資料來往後算 30 分鐘，而在 Redis 中，每個感測器的最新資料時間也不會相同，因為會有些停機或者遺失資料的狀態發生，這樣也方便計算<code>錯誤狀態</code>。</p>
<p>除此之外，由於取到的資料是每 30 分鐘計算，也要特別檢查這 30 分鐘中，有沒有遺失的資料，比方說:</p>
<p><code>2021 08/22 07:42</code>
<code>2021 08/22 07:46</code>
<code>2021 08/22 07:48</code>
<code>2021 08/22 07:50</code></p>
<p>少了 <code>07:44</code> 的資料，這個時候就要補上一筆空的資料，並標記為遺失資料，等待晚上補資料的程式運作，再根據遺失的資料再去 api 取資料。</p>
<h3>處理與計算</h3>
<p>這部分包含</p>
<ul>
<li>擷取重要資訊與數據</li>
<li>單位換算</li>
<li>去除不正常數據</li>
</ul>
<p>取到的感測器訊息會是含有許多不同的參數，但有時候不需要這麼多參數，所以只取得我們需要的數據，然後做單位換算，或者是合成下一步<code>計算</code>需要的數據。</p>
<p>最後就是偵測一些不正常的數據，比方說早上 6~9 點的日射量不會 <code>>15000</code>，超過就需要懷疑是否裝置問題，還有就是環境溫度不得 <code>>39</code>，過高也有問題。</p>
<p>計算數據的話，也沒什麼問題，就是計算類似</p>
<ul>
<li>錯誤狀態</li>
<li>總體數據</li>
<li>排名</li>
<li>預期達成數據 (參考發電)</li>
</ul>
<p>其中比較難算的是 <code>參考發電</code>，畢竟需要各個感測器正常運作的數值加上溫度及日射計的正常運作，且從早上第一筆開始累計，也就是說若有遺失或損毀的部分，我們的程式設計機制就是將前一筆資料作為依據，複製到後一筆做為參考，並把這筆資料標記為遺失待補。</p>
<p>也就是說:</p>
<p><code>07:42</code> 早上日射量為 <code>231</code>
<code>07:44</code> 為 <code>213</code>
<code>07:46</code> 的當下缺了資料，那就把上一筆的 <code>213</code>作為這兩分鐘偵測到的日射量。</p>
<h3>儲存與發佈</h3>
<ul>
<li>儲存
<ul>
<li>將原始資料及計算後的數據儲存於資料庫</li>
<li>將初始化值，需更新的值及最終計算結果儲存於 Redis</li>
</ul>
</li>
<li>透過 MQTT 發布給前端</li>
</ul>
<p>原則上沒什麼太大的問題，就是照這樣的步驟去實作</p>
<h2>回補資料 (Data Patcher)</h2>
<p>由於 IoT 裝置會時常發現不穩定的狀況，所以有時候資料會遺失或者需要停機維修，這時候後端的平台會接收不到資訊，而造成計算出來的數據與真實的狀況可能有差異。</p>
<p>為了盡量能夠貼近更真實的數據，我們也同樣實作了一個補資料的程式(Data Patcher)，在晚上凌晨時啟動，用整個晚上的時間重新取得過去遺失的資料，然後重新計算像是<code>年/月累積參考發電或實際發電</code>這種需要累加的數據，再更新回資料庫。</p>
<p><img src="https://mingjungyu.files.wordpress.com/2021/09/screen-shot-2021-09-25-at-4.52.53-pm.png" alt=""></p>
<p>所以架構圖會如上方，晚上太陽能廠沒有在運作，人們也需要休息，所以前端網頁不需要更新資訊。</p>
<p>程式步驟如下：</p>
<ul>
<li>從資料庫取得被標記為<code>遺失</code>的資料</li>
<li>透過資料的<code>裝置id</code>，找到 <code>裝置資訊</code></li>
<li>組成 <code>裝置資訊</code> + <code>遺失時間</code>，並向 api 取資料</li>
<li>處理資料</li>
<li>重新計算資料 (像是那種累積性質的資料，修正資料正確性)</li>
<li>儲存回資料庫跟更新 Redis</li>
</ul>
<p>補資料跟早上取資料的程式邏輯差不多，只是欲取的資料從資料庫中被標記為<code>遺失</code>的資料為主。</p>
<p>計算的部分，因為只要有資料補回，就會影響像是:</p>
<ul>
<li><code>年參考發電量</code></li>
<li><code>月參考發電量</code></li>
<li><code>總實際發電量</code></li>
</ul>
<p>類似這類型的數據指標，而且別忘了剛剛有說過像是<code>參考發電</code>的數據，必須是該太陽能廠所有的變流器 + 日射計 + 溫度計，從早上開機開始一直累積。</p>
<p>所以只要一有補任何的資料，就要當日全部從凌晨開機的時間點，重算到傍晚關機時間，然後再將每天的數據加總才會是正確的<code>月跟年參考發電</code>，取資料加上重算資料需要一整個晚上的時間也不為過。</p>
<h1>使用到的技術</h1>
<p>對比於上一個路燈的專案，比較中規中矩適合初階者練習，在太陽能監控平台這個專案，基於有先前路燈的經驗，規劃專案的能力更是提升許多，尤其是要規劃程式的架構跟功能分割，已經有先前的能力，在這個專案中，更能熟練的思考。</p>
<p>相比於上一個路燈專案是學會用物件導向實作專案，在此專案最大的進步應該就是終於了解資料結構跟如何運用於專案中增加效能跟讓功能精緻化。</p>
<p>學到的技術包括像是用 Redis，Queue 資料結構跟字典 Dictionary 暫存來儲存一些資訊，提高機動性，效能跟減少大量對資料庫的操作動作。</p>
<p>再來在之後重新做了一個資料收集的程式，也學習到利用 Async 跟 Thread 等模式來提高收集資料的效能。</p>
<p>除此之外，除了對 MQTT 較熟練之外，還初次接觸到了感測器傳來貼近機械語言的訊息，需要做切割跟翻譯，才能讓後續的程式做處理。</p>
<p>學習到跟技術無關的知識則是有請太陽能能源專家讓我們學習到整個太陽能廠的運作，還有有關於太陽能能源的知識，像是如何發電，與太陽的傾斜角度相關，相關的太陽能能源公式，都有學習到。</p>
<p>在這整個太陽能專案中，我認為最主要進步的是那些抽象與邏輯的思維模式，像是</p>
<ul>
<li>學會繪製流程圖</li>
<li>學會規劃程式架構</li>
<li>學會去發想一些後續的防範機制</li>
<li>學會去針對商業邏輯的部份想出優化跟解決機制</li>
</ul>
<p>而我也認為，後端工程師的後續成長，比較像是偏抽象與邏輯思維的鍛鍊，簡單來說就是</p>
<p><strong>怎麼將生活化或商業價值邏輯化，然後實作出來</strong></p>
<p>至於用什麼程式語言或者用什麼套件，都是基本的配備，因為只要邏輯對了，用什麼程式語言或套件都能建構出專案/產品。</p>
<h1>其他</h1>
<h2>特殊的資料庫設計</h2>
<p>在實際的太陽能廠，是有很多個變流器，每個變流器都會記錄他的累積總發電量，但是難免會遇到需要將變流器換掉的機會，新的變流器會將舊的變流器的發電量承接上來，這設計應該是沒什麼問題。</p>
<p>但是在後端程式的資料庫，新的變流器跟舊的變流器理論上是屬於 <code>兩個不同的變流器id</code>，他在太陽能廠的位置是同一個，而且替換時，廠商會直接累計舊的發電量。</p>
<p>不過在程式上，就會是兩個變流器 id，分別的兩個累積量，因此在設計上會需要有一個抽象的思考，本來的思考層面是停在<code>以裝置id做單位</code>，只收集各個 id 的資料，但因為要顧及會有更換裝置，還是需要累積發電量這樣的問題。</p>
<p>因此在資料庫的設計上，會多一個抽象的資料表(Table)，我們以<code>太陽能廠內部架構</code>的思維去設計，也就是說假設內部架構代表<code>D1</code>的變流器<code>DPX0002</code>壞掉了，換了一台<code>DPX3302</code>，也是在同個<code>D1</code>區做互換，而累積的發電量就記在<code>D1</code>這一筆資料中，所以:</p>
<p>在抽象思維的資料表就會記為</p>
<pre><code>id place accumulate_power
1    D1        33402
</code></pre>
<p>然後會有另外一個紀錄裝置的資料表</p>
<pre><code>id device_id place_id accumulate_power
1   DPX0002      D1         30202
2   DPX3302      D1          3200
</code></pre>
<p>所以類似這類型的問題，在這次的太陽能專案會時常遇到。</p>
<h2>太陽能二版資料收集程式</h2>
<p>由於即時資料是每兩分鐘一次，所以照道理來說，我們必須要在 2 分鐘內做到</p>
<ul>
<li>取資料</li>
<li>處理</li>
<li>計算</li>
</ul>
<p>通常會在取資料的部分花費比較多的時間，因為是使用了單一 Thread 機制，也就是說</p>
<p>總共有 116 太陽能場共千多個感測器，然後初版的資料收集程式是用可愛的 for 迴圈，從第一個案場開始取，開始丟對方公司的 api，取完所有的太陽能廠資料後，在進入到處理的部分。</p>
<p>有時候整個程式執行下來，會逼近 1 分 30 秒，如果遇到一些網路不佳的問題，可能會有遺失或 delay 的問題。</p>
<p>因此，我做了第二版的資料收集程式，其實當初也是請求主管能讓我有個專案練習多執行緒 XD，因為我想學，所以這個程式就是拿來學習的成果。</p>
<p>基本上，這個程式在實作時，有學習到了怎麼使用 Python 去實作<code>多執行緒</code>跟 <code>Async</code> 的概念，最終完成的程式是使用<code>Async</code>去實作，也就是說盤點 116 太陽能場，然後分成為<code>116</code>個 task 任務，並使用<code>Async</code>的方式分別去取 api 資料，然後處理，等到最後都彙整完畢在做計算。</p>
<p>改良後的資料數據程式速度大約為 15~30 秒左右，相比第一版的 1 分 30 秒已經快許多，快到我都在懷疑他到底有沒有取到資料 XD</p>
<h2>自己公司去承包太陽能廠佈置與數據</h2>
<p>當初製作這個一頁式監控網站時，其實就是公司希望能夠用自己的力量掌控整個太陽能廠的佈置與監控。</p>
<p>所以其實在我在職時比較靠後面的時間，其中一個太陽能廠就是自己包辦。</p>
<p>雖然我沒有實際參與在其中，但是我看到我的主管跟另外一個能源博士每天都在比價不同的變流器廠牌，然後我主管還要和硬體工程師去合作撰寫軟體然後嵌入感測器中，所有的事情都是第一次遇到也是第一次包辦，所以看得出來他們非常的疲累與急迫，上層給的人又非常少。</p>
<p>其中一個篇軟體方面遇到的問題就是，數據真的是從 MQTT 傳來的，但是由於沒有太多時間做溝通，所以硬體工程師就直接寫成都是<code>01</code>且用一些特殊符號做區段，所以後續再接資料的時候，必須依據從第幾個位元到第幾個位元代表發電量，類似這樣的邏輯去處理從 MQTT 收到的資訊。</p>
<p>老實說，我敢相信如果公司再多給一些人力，一定能夠將承包太陽能廠的佈置跟監控平台的工作給整個包下來並成立部門持續運作新能源廠。</p>
<p>但是試問哪個老闆願意這樣呢？</p>
<p>因為要知道光是蓋一個太陽能廠就直接扔掉了 3-4 億，要靠發電來賺錢，通常需要運作到第七年之後才會陸續賺錢，無法立即看到價值，又毫無效率賺錢，哪個老闆想做這種賠錢的生意呢？</p>
<p>為了國家發展？慈善？</p>
<p>所以，在新官上任後，決策轉彎後，監控平台建置的軟體部門也轉型了，不再做開發了，太陽能廠也從原本想自己承包，改為由外部承包然後把太陽能廠當作商品賣掉，而不是透過後續發電來賺錢。</p>
<p>我仍然相信能源是極其重要，尤其是再生能源，既然每天都會有太陽，利用太陽發電後，供電給人民生活是一個很好選擇，但是很少人知道背後的建置需要花費高額金錢跟多少人力以及帶來的污染都無法細數。</p>
